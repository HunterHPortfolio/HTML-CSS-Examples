Group 1 - CCMP Encapsulation

STAUT WPA2 Test 1.1.1 - CCMP MIC Verification
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.1.1 Part a

This test is actually quite simple, if the STAUT was unable to do any of the things listed, the Ping would be unable to be decrypted or responded to.
Therefore, we just need to check that the STAUT responds to encrypted Pings as well as containing the eight octets of CCMP information.
Same for all parts a-e.

STAUT WPA2 Test 1.1.2 - CCMP Header Format
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.1.2 Part a

For this one we are just seeing if it adds the 8 bytes for the CCMP header, all we have to check for is that MSDU1-MSDU3 contain the CCMP header

Test 1.1.2 Parts b and c

Part b
requires you to go a little deeper into the replies for MSDU1-MSDU3.
Basically, we need to check that the STAUT sets the Ext IV bit to 1 in each Ping reply.
If you highlight the CCMP header, it shows you the octets within the frame.
The Key IV is in the fourth octet (highlighted in red).
This value (20) is in hex but if you were to convert it to binary, it would be 0010 0000,
the Key IV is the 6th bit starting from the right, we clearly see it set to 1!
Part c
since we are looking in the same exact place as b, I have grouped these cases together
the reserved bits are contained in the fourth octet as well, for this to pass - the first 5 bits in the above value (0010 0000) should be set to 0. Both cases are a pass if the 3rd and fourth octets equal: 00 20.

Test 1.1.2 Part d

For this part, we just need to check that the Packet Number for each MPDU transmitted is incrimented.

STAUT WPA2 Test 1.1.3 - CCMP Encryption Verification
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.1.3 Parts a and b

This one is pretty easy, as long as the STAUT responds to each frame then it's able to decrypt these frames.


Group 2 - CCMP Decapsulation

STAUT WPA2 Test 1.2.1 - CCMP MIC Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2, 11.4.3.3, and 11.4.3.4.3

Test 1.2.1 Part a

For this part, the STAUT should discard frames MSDU1-MSDU3 and MSDU5
It is ok for the STAUT to acknowledge these frames

STAUT WPA2 Test 1.2.2 - CCMP Header Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.2.2 Part a

The STAUT should accept keys for each Key ID defigned, this just means that no frame should be discarded due to an unrecognized Key ID.

Test 1.2.2 Part b

The APUT should ignore reserved bits (b0-b5) of the 4th octet and all bits of the 3rd octet in the CCMP header
we can see that this is a pass because the APUT ignored and successfully transmitted an ICMP Echo Reply to MSDU1 (frame 181)

Test 1.2.2 Part c

This is a Failure Case
The APUT should discard CCMP encrypted frames with the Extended IV bit set to 0 (data frame 197).
It is ok for the APUT to Acknowledge this frame but here we can see the APUT sent an ICMP Echo Reply to it.

Test 1.2.2 Part d

This case is Informational
The APUT may or may not discard any frame containing an invalid Key ID.
in this case, the frame (frame 213) is acknowledged and then discarded

STAUT WPA2 Test 1.2.3 - CCMP Decryption Verification
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.4

Test 1.2.3 Part a

For this part, the STAUT should be able to decrypt, and respond to each frame (MSDU1-MSDU4).
as you see here the STAUT does just that.

Test 1.2.3 Part b

For this part, the STAUT should not be able to decrypt or respond to MSDU5.

Test 1.2.3 Part c

The STAUT should behave exactly like 1.2.3 Part b, frames MSDU6-MSDU11 should be discarded

STAUT WPA2 Test 1.2.4 - CCMP PN Replay Protection
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.2.4 Part a

We need to make sure that for MSDU1, MSDU3, and MSDU5 the Packet Number (located in the CCMP Ext. Vector) is incrimented ONLY for these frames and MSDU2 and MSDU4 are not being responded to and/or effect the PN count for the aformentioned MSDUs
These frames are represented as data frames in this particular case but normally
it would be represented as Ping Requests/Responses

Test 1.2.4 Part b

This is a failure case
The STAUT is supposed to detect replayed frames and discard MSDUs whose constituent MPDU PN value isn't sequential (MSDU7, MSDU10, MSDU13, MSDU16).
In this case, the PN for the first Echo (ping) Request ends with 56 followed by an Echo (ping) Request whose PN ends with 10057.
since the STAUT responded to MSDU8 and MSDU9(with non sequential PN values) this is a fail

Test 1.2.4 Part c

The STAUT should use the PN from the recieved MPDU to detect replayed frames and should not be discarded.


Group 3 - EAPoL-Key Reception

STAUT WPA2 Test 1.3.1 - Descriptor Type Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.1 Part a

For this part to Pass, we just need to make sure the 4-way handshake goes through successfully

Test 1.3.1 Part b

This is a failure case
The STAUT is supposed to silently discard MSDU1-MSDU6. As you see here, the STAUT acknowledges the frame which makes it a fail.

STAUT WPA2 Test 1.3.2 - Key Information Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.2 Part a

For MSDU1-MSDU10, the STAUT should successfully complete the 4-way handshake (thus ignoring the reserved bits)
Note: MSDU3 and MSDU8 contain bit 13, which is no longer reserved.

Test 1.3.2 Part b

This is a failure case
For MSDU11-MSDU22, the STAUT should silently discard these frames, here the STAUT acknowledges MSDU12

Test 1.3.2 Part c

The STAUT should silently discard MSDU23
This part is frequently failed

STAUT WPA2 Test 1.3.3 - Key Length Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.3 Part a

This is a Failure case
The APUT should silently discard MSDU1-MSDU5
As you see here, the APUT acknowledges MSDU1 (frame 215) and even completes the 4-way handshake which is a fail.

Test 1.3.3 Part b

This is a Failure case
The APUT should silently discard MSDU6-MSDU10
As you see here, the APUT acknowledges MSDU7 (frame 1785) which is a fail.

STAUT WPA2 Test 1.3.4 - Key Replay Counter Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.4 Part a

The STAUT should use the same replay counter as the key before it.

Test 1.3.4 Part b

This is a failure case
The STAUT should silently discard any EAPoL-Key frames recieved with a Key Replay Counter field that is less than or equal to any recieved in a valid message and not successfully complete the 4-way handshake

STAUT WPA2 Test 1.3.5 - Key Nonce Field Processing
IEEE Std 802.11™-2012 Edition, Subclause and IEEE Std 802.1x™-2004 Edition, Subclause

Test 1.3.5 Part a and b

The STAUT should not be able to complete the 4-way handshake.

STAUT WPA2 Test 1.3.6 - EAPoL-Key IV Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.6 Part a

This is a Failure case
The APUT should silently discard MSDU1 and MSDU2
if the APUT only sends an Acknowledgment it would still fail due to it not being silent

STAUT WPA2 Test 1.3.7 - Key RSC Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.7 Part a

This is a Failure case
The STAUT should silently discard MSDU1, as you see here the STAUT successfully completed a 4-way handshake using MSDU1.

Test 1.3.7 Part b

This is an Informative Test
The DUT may or may not ignore the unused octets of the Key RSC. Observe what it does. As you see here, the STAUT ignored the unused octets and successfully completed the 4-way handshake.

Test 1.3.7 Part c

The STAUT should discard MSDU4, as you see here, the TS resends MSDU4 and the STAUT does not process it at all.

STAUT WPA2 Test 1.3.8 - Reserved Octets Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.8 Part a

This test we are checking that the APUT ignores reserved bits within EAPoL-Key frames
We just need to check that the APUT successfully completes the 4-way hadshake for MSDU1 and MSDU2

STAUT WPA2 Test 1.3.9 - Key MIC Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.9 Part a

This is a Failure case
The STAUT should silently discard MSDU2 and MSDU3, as you see here the STAUT acknowledges MSDU2

STAUT WPA2 Test 1.3.10 - Key Data Length Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.10 Part a

The STAUT should receive EAPoL-Key frames with invalid Key Data Lengths without failure. The STAUT should discard MSDU1 and MSDU2

Test 1.3.10 Part a

The STAUT should ignore any pad bytes appended to the Key Data field and included in the Key Data Length field. If the 4-way handshake goes through then we know that the STAUT successfully ignored the pad bytes.

STAUT WPA2 Test 1.3.11 - Key Data Field Processing (Pairwise Message 1)
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.11 Part a

For this part, the APUT just needs to ignore any IEs that are unkown (MSDU1 and MSDU2) and should successfully complete the 4-way handshake.

Test 1.3.11 Part b

Similiar to 1.3.11a
The APUT just needs to ignore any pad bytes appended to an Encryped Key Data field (MSDU3)

Test 1.3.11 Part c

The STAUT should accept incorrect PMKIDs (MSDU4 and MSDU5)

STAUT WPA2 Test 1.3.12 - Key Data Field Processing (Pairwise Message 3)
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.12 Part a

This is a Failure case
The STAUT should discard MSDU2, as you see here the 4-way handshake is instead successfully completed.

Test 1.3.12 Part b

The STAUT should discard MSDU3 and MSDU4

Test 1.3.12 Part c

The STAUT should ignore any extraneous IEs or unknown KDEs (MSDU5 and MSDU6). As you see here, the STAUT ignores these valuse and continues the 4-way handshake.

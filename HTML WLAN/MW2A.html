Group 1 - CCMP Encapsulation

APUT WPA2 Test 1.1.1 - CCMP MIC Verification
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.1.1 Part a-e

This test is actually quite simple, if the APUT was unable to do any of parts listed, the Ping would not able to be to be decrypted or responded to.
Therefore, we just need to check that the APUT responds to encrypted Pings as well as containing the eight octets of CCMP information.
Same for all parts a-e.

APUT WPA2 Test 1.1.2 - CCMP Header Format
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.1.2 Part a

For this one we are just seeing if it adds the 8 bytes for the CCMP header, all we have to check for is that MSDU1-MSDU3 contain the CCMP header

Test 1.1.2 Parts b and c

Part b
This requires you to go a little deeper into the ICMP Echo Replies for MSDU1-MSDU3.
Basically, we just need to check that the APUT sets the Ext IV bit to 1 in each Ping reply.
If you highlight the CCMP header, it shows you the octets within the frame.
The Key IV is in the fourth octet (highlighted in red).
This value (20) is in hex but if you were to convert it to binary, it would be 0010 0000,
the Key IV is the 6th bit starting from the right, we clearly see it set to 1! If this value is 20 then this is a Pass
Part c
since we are looking in the same exact place as b, I have grouped these two cases together
the reserved bits are contained in the fourth octet as well, for this to pass - the first 5 bits in the above value (0010 0000) should be set to 0. Both cases are a pass if the 3rd and fourth octets equal: 00 20.

Test 1.1.2 Part d

For this part, we just need to check that the Packet Number for each MPDU transmitted is incrimented.

APUT WPA2 Test 1.1.3 - CCMP Encryption Verification
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.1.3 Parts a-d

This one is pretty easy, as long as the APUT responds to each frame it means it's able to decrypt the frames.


Group 2 - CCMP Decapsulation

APUT WPA2 Test 1.2.1 - CCMP MIC Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2, 11.4.3.3, and 11.4.3.4.3

Test 1.2.1 Part a

The APUT should discard MPDUs with invalid MICs: MSDU1, MSDU2, and MSDU4

APUT WPA2 Test 1.2.2 - CCMP Header Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.2.2 Part a

The APUT should ignore reserved bits (b0-b5) of the 4th octet and all bits of the 3rd octet in the CCMP header
we can see that this is a pass because the APUT ignored and successfully transmitted an ICMP Echo Reply to MSDU1 (frame 181)

Test 1.2.2 Part b

This is a Failure Case
The APUT should discard CCMP encrypted frames with the Extended IV bit set to 0 (data frame 197).
It is ok for the APUT to Acknowledge this frame but here we can see the APUT sent an ICMP Echo Reply to it.

Test 1.2.2 Part c

This case is Informational
The APUT may or may not discard any frame containing an invalid Key ID.
in this case, the frame (frame 213) is acknowledged and then discarded

APUT WPA2 Test 1.2.3 - CCMP Decryption Verification
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.4

Test 1.2.3 Part a

For this part, the APUT should be able to decrypt, and respond to each frame MSDU1-MSDU4.
As you see here the APUT does just that.

Test 1.2.3 Part b

For this part, the APUT should not be able to decrypt or respond to MSDU5.

Test 1.2.3 Part c

The APUT should behave exactly like 1.2.3 Part b, frames MSDU6-MSDU11 should be discarded as well

APUT WPA2 Test 1.2.4 - CCMP PN Replay Protection
IEEE Std 802.11™-2012 Edition, Subclause 11.4.3.2 and 11.4.3.3

Test 1.2.4 Part a

We need to make sure that for MSDU1, MSDU3, and MSDU5 the Packet Number (located in the CCMP Ext. Vector field) is incrimented ONLY for these frames while MSDU2 and MSDU4 are not responded to and/or effect the PN count for the aformentioned MSDUs.
These frames are represented as data frames in this particular case but normally
it would be represented as Ping Requests/Responses

Test 1.2.4 Part b

This is a failure case
The APUT is supposed to detect replayed frames and discard MSDUs whose constituent MPDU PN value isn't sequential (MSDU7, MSDU10, MSDU13, MSDU16).
In this case, the PN for the first ICMP Echo Request ends with 56 followed by an ICMP Echo Request whose PN ends with 10057.
since the APUT responded to MSDU8 and MSDU9(with non sequential PN values) this is a fail


Group 3 - EAPoL-Key Reception

APUT WPA2 Test 1.3.1 - Descriptor Type Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.1 Part a

For this part to Pass, we just need to make sure the 4-way handshake goes through successfully

Test 1.3.1 Part b

This is a failure case
The APUT should silently discard invalid EAPoL-Key frames
As you see here, The APUT acknowledges MSDU1 (frame 196) which makes it a fail.

APUT WPA2 Test 1.3.2 - Key Information Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.2 Part a

For MSDU1-MSDU10, the APUT should successfully complete the 4-way handshake (thus ignoring the reserved bits)
Note: MSDU3 and MSDU8 contain bit 13, which is no longer reserved.

Test 1.3.2 Part b

This is a Failure case
The APUT should silently discard all EAPoL-Key frames containing invalid Key Descriptor Versions (MSDU11-MSDU22), the APUT should not be able to successfully complete the 4-way handshake.

Test 1.3.2 Part c

The APUT should silently discard MSDU23

Test 1.3.3 Part a

This is a Failure case
The APUT should silently discard MSDU1-MSDU5
As you see here, the APUT acknowledges MSDU1 (frame 215) and even completes the 4-way handshake which is a fail.

Test 1.3.3 Part b

This is a Failure case
The APUT should silently discard MSDU6-MSDU10
As you see here, the APUT acknowledges MSDU7 (frame 1785) which is a fail.

APUT WPA2 Test 1.3.4 - Key Nonce Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.4 Part a

This is a failure case
The APUT should silently discard MSDU1. In this case, the APUT acknowledged MSDU1 (frame 357) which means this is a fail.

Test 1.3.4 Part b

The APUT should not successfully complete the 4-way handshake
As you see here the APUT didn't process MSDU2 (frame 608 and 624). Seeing that the TS retransmitted EAPoL-Key #3 we can tell it wasn't successful
You can also determine this by an unsuccessful the Echo response/reply afterwards.

APUT WPA2 Test 1.3.5 - Key Replay Counter Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, 11.6.6.5, and 11.6.5 (informative) and and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.5 Part a and b

There's 2 parts to this trace
Part a - the APUT should recieve MSDU1 without the failing
Part b - the APUT should not successfully complete the 4-way handshake upon reception of MSDU2 - this might be tricky but a little bit further down the trace a Ping request is sent and the APUT doesn't respond meaning it didn't successfully complete the 4-way handshake

APUT WPA2 Test 1.3.6 - EAPoL-Key IV Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.6 Part a

This is a Failure case
The APUT should silently discard MSDU1 and MSDU2
if the APUT only sends an Acknowledgment it would still fail due to it not being silent

APUT WPA2 Test 1.3.7 - Key RSC Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.7 Part a

This is a Failure case
MSDU1 and MSDU2 should be silently discarded

APUT WPA2 Test 1.3.8 - Reserved Octets Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.8 Part a

This test we are checking that the APUT ignores reserved bits within EAPoL-Key frames
We just need to check that the APUT successfully completes the 4-way hadshake for MSDU1 and MSDU2

APUT WPA2 Test 1.3.9 - Key MIC Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2, 11.6.6.3, and 11.6.6.5 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.9 Part a

This is a Failure Case
The APUT should silently discard MSDU1-MSDU4

APUT WPA2 Test 1.3.10 - Key Data Length Field Processing
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.10 Part a

The APUT should recieve MSDU1 and MSDU2 without failure
it doesn't neccessarily need to complete the 4-way handshake

APUT WPA2 Test 1.3.11 - Key Data Field Processing (Pairwise Message 2)
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.11 Part a

For this part, the APUT just needs to ignore any IEs that are unkown (MSDU1)

Test 1.3.11 Part b

Similiar to 1.3.11a
The APUT just needs to ignore any pad bytes appended to an Encryped Key Data field (MSDU2)

Test 1.3.11 Part c

The APUT should terminate the association upon reception of any frame that has an RSN IE field that doesn't bitwise match the RSN IE contained in the association or reassociation request frame prior (MSDU3-MSDU6).

APUT WPA2 Test 1.3.12 - Key Data Field Processing (Pairwise Message 4)
IEEE Std 802.11™-2012 Edition, Subclause 11.6.2 and IEEE Std 802.1x™-2004 Edition, Subclause 7.6

Test 1.3.12 Part a

The APUT should accept EAPoL-Key frames with empty data fields
As you see here, the APUT accepts MSDU1 (frame 167)
we can verify that the 4-way handshake was successful by looking at pings that follow the Authentication.

Test 1.3.12 Part b

The APUT should recieve MSDU2 without failure
pretty vague but as long as the APUT continues to broadcast then it's a pass.

Test 1.3.12 Part c

This is a Failure Case
The APUT should ignore any IEs or KDEs that are uknown
As you see here, the APUT does not successfully ignore it, we can see that the APUT retries the authentication by re-sending Key #3 signifying that MSDU3 (frame 568 and 583) was not successfully processed

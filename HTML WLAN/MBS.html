STAUT Base Test 1.1.3 - Deauthentication Processing
IEEE Std 802.11™-2012 Edition, Subclauses 10.3, 4.5, 8.4.1.7, 9.3.2.2, and Annex J

Test 1.1.3 Part a

The STAUT should not pass either Class 2 or 3 traffic after processing MSDU1 (Deauthentication frame 347).
As you see here, the STAUT acknowledges MSDU1 on frame 348 (processing it),
the TS then sends an Echo (ping) request on frame 393, and lastly the STAUT Deauthenticates on frame 395 due to class 3 traffic being sent.

Test 1.1.3 Part b

This Test is Informational Only
Once The STAUT Recieves MSDU2 (Deauthentication frame 3424),
Observe How The STAUT Responds to the frame.
(Does the STAUT stop responding, does it process the Frame, or does it completely ignore it?).
As you see here, This is an example of MSDU2 not being processed.
We know this because of the successful Echo (ping) Request/Reply following it.

Test 1.1.3 Part c

The STAUT should not acknowledge MSDU3 (Deauthentication frame 3695),
it also should Not Be Processed. It is okay for the frame to repeat like this.
As you see here, it is not responded to and is followed by a successful Echo (ping) Request/Reply
further indicating that the frame wasn't processed.

Group 2 - RTS and Fragmentation

STAUT Base Test 1.2.1 - Recovery Procedure and Retransmit Limits
IEEE Std 802.11™-2012 Edition, Subclause 9.3.4.4

Test 1.2.1 Part a

Here you will see a bunch of Request-to-send (RTS) frames
in response to MSDU2 (frame 216)
the value of the dot11ShortRetryLimit and aLongRetryLimit may not be known,
in which case the consistency of the number of retries needs to be checked.
Each Request-to-send frame should have the retry bit set to 0

Test 1.2.1 Part b

Here you will see a bunch of ICMP Echo Replies in response to MSDU2 (frame 380).
The value of the dot11ShortRetryLimit and aLongRetryLimit may not be known,
in which case the consistency of the number of retries needs to be checked.

Test 1.2.1 Part c

Here you will also see a bunch of ICMP Echo Replies in response to MSDU1 (frame 574).
The value of the dot11ShortRetryLimit and aLongRetryLimit may not be known,
in which case the consistency of the number of retries needs to be checked.

Test 1.2.1 Part d

Here you will see a lot of Fragmented Frames in response to the ICMP Echo Request (Frame 203). The STAUT should not attempt to transmit any other fragments than the first fragment
The value of the dot11ShortRetryLimit and aLongRetryLimit may not be known,
in which case the consistency of the number of retries needs to be checked.

STAUT Base Test 1.2.2 - RTS/CTS and Directed MPDU Transfer
IEEE Std 802.11™-2012 Edition, Subclauses 9.3.2.5, 9.3.2.6, 9.3.5, and 8.3.1

Test 1.2.2 part a

There are 3 parts to this:
1. The STAUT should not use RTS/CTS prior to the ICMP Echo Response to MSDU1 (frame 267)
2. The STAUT should not use RTS/CTS prior to the ICMP Echo Response to MSDU2 (frame 281)
3. The STAUT should use RTS/CTS prior to the ICMP Echo Response to MSDU3 (frame 298) - RTS/CTS on frames 296-297.

Test 1.2.2 Part b

This is the same as 1.2.2a - There are 3 parts to this:
1. The STAUT should not use RTS/CTS prior to the ICMP Echo Response to MSDU5 (frame 139)
2. The STAUT should not use RTS/CTS prior to the ICMP Echo Response to MSDU6 (frame 153)
3. The STAUT should use RTS/CTS prior to the ICMP Echo Response to MSDU7 (frame 170) - RTS/CTS on frames 168-169.

Test 1.2.2 Part c

The STAUT should ONLY use RTS/CTS prior to the first fragment for the ICMP Echo response to MSDU4 (Fragmented frame 327)

Test 1.2.2 Part d

Currently the script is sending MSDU4 instead of MSDU8, This is essentially testing the same thing as 1.2.2 Part b though - this is a pass

STAUT Base Test 1.2.3 - Broadcast and Multicast MPDU Transfer
IEEE Std 802.11™-2012 Edition, Subclause 9.3.6

Test 1.2.3 Part a

Few things to check off here
The STAUT uses CTS/RTS before it sends the ICMP Echo Response.
The more fragments bit is set in all but the last fragment
correctly set the destination address
same sequence number in all fragments
finally an incrementing fragment number

Test 1.2.3 Part b

This one is pretty simple
we need to check that the size of each fragment is even and that it isn't larger than the set threshold (399 for this particular trace)

STAUT Base Test 1.2.4 - Multirate Support
IEEE Std 802.11™-2012 Edition, Subclause 9.7

Test 1.2.4 Parts a-f

CHECK THIS DESCRIPTION This test sounds complicated but is pretty simple, we are just checking that the STAUT transmits Control response frames (CTS and ACK frames) at one of the basic rates in the BSSBasicRateSet at a rate less than or equal to the rate of the previous frame sent.
For Part a-c
Basic Rates = 1 and 2
Extended Rates = 5.5 and 11
For Parts d-f
Basic Rates = 6, 12, and 24
Extended Rates = 9, 18, 36, 48, and 54
